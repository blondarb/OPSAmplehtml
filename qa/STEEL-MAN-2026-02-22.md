## Executive Summary

The AI Triage Tool is a well-architected clinical decision support system with a sound deterministic scoring approach. The two-stage extraction pipeline (Phase 2) adds genuine clinical value by enabling triage of non-referral note types. **However, the strongest argument against this product being ready is the complete absence of authentication and authorization** — any anonymous internet user can submit triage requests consuming OpenAI API credits, override any triage result, and read/modify any session. Additionally, the batch processing feature is UI-complete but functionally incomplete (only first file is processed), and there are several input validation gaps that could surface in real clinical use.

**Overall Readiness:** Ready with Caveats (for demo/internal use only; not production-ready for clinical deployment)

## PRD & Requirements Gaps

| # | Gap | Severity | Impact | Recommendation |
|---|-----|----------|--------|----------------|
| R1 | No authentication requirement specified for triage endpoints | S1 | Anyone on the internet can consume OpenAI credits and access triage functionality. In production, unauthorized access to a clinical decision tool creates liability. | Add auth middleware to all triage API routes. At minimum, require a valid Supabase session token. |
| R2 | Pediatric patient handling undefined | S2 | System prompt says "ADULT (≥18 years)" but no code-level validation exists. An 8-year-old patient submission returns `insufficient_data` via AI discretion, not deterministic validation. AI behavior here is non-deterministic and could vary. | Add explicit age validation: if `patient_age < 18`, return 400 with "This system is designed for adult patients (18+). Pediatric referrals should be processed through your pediatric neurology pathway." |
| R3 | Batch processing requirements incomplete | S2 | UI accepts up to 20 files, but code only processes `files[0]`. No PRD section specifies batch orchestration behavior (sequential vs parallel, failure handling, partial results). | Either remove multi-file upload UI or implement sequential processing. The `BatchItem` and `BatchState` types exist but are unused. |
| R4 | No data retention or deletion policy | S3 | Triage sessions are stored indefinitely in Supabase. No documented retention period, no delete mechanism, no data export. PHI concerns if real patient data is used. | Define a retention policy (e.g., 90 days for demo). Add a session deletion endpoint. |
| R5 | No offline/degraded mode behavior defined | S3 | If OpenAI API is down, the tool returns a 500 error. No cached fallback, no manual triage workflow guidance. | Show a clear "AI unavailable" state with instructions for manual triage. |
| R6 | Override audit trail lacks detail | S3 | Override stores category but no free-text reason field. The 5 predefined categories may not capture nuanced clinical reasoning. | Add optional free-text field for override justification. |
| R7 | No concurrent user/rate limit requirements specified | S2 | No rate limiting on any endpoint. A malicious or misconfigured client could exhaust OpenAI API quota rapidly. | Add rate limiting (e.g., 10 requests/minute per IP for unauthenticated, 30/min for authenticated). |

## Architecture & Code Findings

| # | Finding | Area | Severity | File(s) | Recommendation |
|---|---------|------|----------|---------|----------------|
| A1 | **No authentication on any triage endpoint** — All 5 API routes (`/api/triage`, `/api/triage/extract`, `/api/triage/fuse`, `/api/triage/[id]/override`, `/api/triage/samples`) accept unauthenticated requests. No middleware, no token check, no session validation. | Security | S1 | `src/app/api/triage/route.ts`, all route files | Add auth middleware. The rest of Sevaro uses Supabase Auth — extend it to triage routes. |
| A2 | **Override endpoint has no ownership check** — Any user can override any triage session by knowing or guessing the UUID. The endpoint just does `.eq('id', id)` with no user/tenant filtering. | Security | S1 | `src/app/api/triage/[id]/override/route.ts:33` | Add session ownership verification: check that the requesting user/tenant owns the session before allowing override. |
| A3 | **RLS policies set to "allow all"** — Both `triage_sessions` and `triage_batches` have `USING (true) WITH CHECK (true)`. This means any Supabase client can read/modify all sessions. | Security | S1 | `supabase/migrations/020_triage_sessions.sql`, `021_triage_enhancements.sql` | Replace with proper RLS policies tied to `auth.uid()` or tenant_id, matching the pattern used for other tables. |
| A4 | **Triage route missing `maxDuration` export** — Extract and fuse routes export `maxDuration = 60`, but the main triage route does not. On Vercel Hobby plan, default is 10s; on Pro, 60s. With gpt-5.2 taking 13-17s, hobby-plan deployments will timeout. | Code | S2 | `src/app/api/triage/route.ts` | Add `export const maxDuration = 60` to the triage route. |
| A5 | **No runtime validation of AI response structure** — AI response is parsed with `JSON.parse(rawContent) as AITriageResponse` — a cast, not validation. If gpt-5.2 returns unexpected structure (missing fields, wrong types, scores outside 1-5), the scoring code will silently produce incorrect results or crash. | Code | S1 | `src/app/api/triage/route.ts:101` | Add Zod or manual validation: verify all 5 dimension scores exist and are integers 1-5, verify boolean fields, verify arrays. Return 500 with retry message if validation fails. |
| A6 | **No max text length validation on triage route** — The extract route truncates at 50K chars, but the triage route accepts arbitrarily long input. A 60K-char input was accepted and processed (sending all 60K chars to gpt-5.2, consuming excessive tokens). | Code | S2 | `src/app/api/triage/route.ts` | Add `FILE_CONSTRAINTS.MAX_TEXT_LENGTH` validation to the triage route, matching the extract route. |
| A7 | **`source_type` hardcoded to `'pdf'` for all uploads** — In `handleApproveExtraction`, `source_type: inputMode === 'upload' ? 'pdf' : 'paste'` always sends `'pdf'` regardless of whether the uploaded file was .docx or .txt. | Code | S2 | `src/app/triage/page.tsx:140` | Use the actual source type from the extraction result: `source_type: extraction?.source_filename?.endsWith('.docx') ? 'docx' : ...` or better, store and pass `extraction.source_type` if available. |
| A8 | **Malformed JSON returns 500 instead of 400** — Sending `not valid json at all` to `/api/triage` returns status 500 with the raw parser error message exposed (`"Unexpected token 'o', \"not valid j\"... is not valid JSON"`). Should be 400 with a clean message. | Code | S3 | `src/app/api/triage/route.ts:181-192` | Wrap `request.json()` in try/catch and return 400 with "Invalid request body" message. |
| A9 | **Batch processing only handles first file** — `handleSubmitFiles` in `page.tsx` does `const file = files[0]` and ignores the rest. The `FileUploadZone` UI happily accepts up to 20 files. This is a misleading UX. | Code | S2 | `src/app/triage/page.tsx:103` | Either implement sequential batch processing (the types and UI already exist) or limit FileUploadZone to 1 file with a "batch mode coming soon" note. |
| A10 | **Extraction results not persisted to database** — Only the final triage result is stored in `triage_sessions`. The extraction itself (note type, confidence, key findings, original text) is held in client state only. If the user refreshes during the review step, extraction is lost. | Code | S2 | `src/app/api/triage/extract/route.ts` | Store extractions in a new `triage_extractions` table or add extraction JSON to the triage_sessions row at extraction time. |
| A11 | **Fusion result not persisted** — The fusion endpoint returns results but stores nothing in the database. The `fusion_group_id` is generated but never written to any table. | Code | S3 | `src/app/api/triage/fuse/route.ts` | Store fusion results in the database, or at minimum log them for audit purposes. |

This key is fetched per-request (no caching) and could be a security concern if RLS is misconfigured. | Security | S3 | Multiple route files | Cache the API key in-memory for the serverless function lifecycle. Consider removing the RPC fallback in production. |

## What's Working Well

1. **Deterministic scoring architecture** — The split between AI-generated raw scores and application-calculated tiers is genuinely excellent. It makes the system auditable, testable, and resistant to AI hallucination on the final decision. This is better than what most clinical AI tools do.

2. **Anti-bias instruction and clinical safety** — The system prompt explicitly instructs the AI not to adjust scoring based on demographics. The emergent override, red flag override, and insufficient data pathways create proper clinical safety nets. The disclaimer text is prominently displayed.

3. **Two-stage extraction pipeline** — The ability to take a 15K-character ED note and extract only the neurology-relevant information before scoring is a genuinely useful clinical innovation. The editable review step gives clinicians control.

4. **Prompt injection resistance** — The AI successfully resisted a direct prompt injection attempt, returning `insufficient_data` rather than following the injected instructions. While this isn't architecturally guaranteed, gpt-5.2's instruction-following robustness is working well here.

5. **Comprehensive output** — The triage result includes clinical reasons in PCP-friendly language, specific workup recommendations, failed therapy tracking, subspecialty routing with rationale, and transparent dimension-by-dimension scoring. This is significantly more useful than a simple "urgent/routine" label.
